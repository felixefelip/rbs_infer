## Parte 1: Class-level attribute accessors

### Problema

A gem enumerize cria métodos de classe com o nome de cada atributo enumerizado. Esses métodos retornam um `Enumerize::Attribute` e são amplamente usados em formulários e lógica de negócio:

```ruby
class Post < ApplicationRecord
  extend Enumerize
  enumerize :status, in: [:draft, :published, :archived]
end

Post.status           # => #<Enumerize::Attribute ...>
Post.status.values    # => ['draft', 'published', 'archived']
Post.status.options   # => [['Draft', 'draft'], ['Published', 'published'], ...]
Post.status.find_value(:draft)  # => #<Enumerize::Value ...>
```

O gerador atual não emite nenhum tipo para esses class methods. Isso causa erros de tipo ao usar `Post.status.options` em views/forms.

### API do `Enumerize::Attribute` (gem_rbs_collection)

A definição em `gem_rbs_collection` já existe e inclui:

```rbs
# .gem_rbs_collection/enumerize/2.5/enumerize-generated.rbs
class Enumerize::Attribute
  attr_reader klass: untyped
  attr_reader name: untyped
  attr_reader values: untyped
  attr_reader default_value: untyped
  attr_reader i18n_scope: untyped
  def find_value: (untyped value) -> (untyped | nil)
  def find_values: (*untyped values) -> untyped
  def each_value: () { (untyped) -> untyped } -> untyped
  def options: (?Hash[untyped, untyped] options) -> untyped
end
```

Portanto, **não precisamos definir `Enumerize::Attribute` no RBS gerado** — ela já existe. Basta emitir os class methods que retornam esse tipo.

### RBS esperado

Para cada `enumerize :attr_name, ...` no modelo, gerar:

```rbs
class ::Post
  # ... (métodos existentes) ...

  def self.status: () -> Enumerize::Attribute
  def self.priority: () -> Enumerize::Attribute
end
```

### Implementação

#### 1. Modificar `build_rbs`

Adicionar uma chamada a um novo método `build_class_attribute_accessor` dentro do loop de `enumerize_calls`:

```ruby
# em build_rbs, dentro do bloco enumerize_calls.each
def build_rbs(class_name, enumerize_calls)
  # ... existente ...
  enumerize_calls.each do |call|
    lines.concat(build_instance_methods(call))
    lines.concat(build_predicate_methods(call))
    lines.concat(build_scope_methods(call))
    lines.concat(build_class_attribute_accessor(call))  # ← novo
  end
  # ...
end
```

#### 2. Novo método `build_class_attribute_accessor`

```ruby
def build_class_attribute_accessor(call)
  ["  def self.#{call[:name]}: () -> Enumerize::Attribute", ""]
end
```

#### 3. Testes esperados

Verificar no RBS gerado do `Post`:

```rbs
def self.status: () -> Enumerize::Attribute
def self.priority: () -> Enumerize::Attribute
def self.category: () -> Enumerize::Attribute
```

Verificar com `steep check` que `Post.status.options` resolve sem erros.

### Impacto

- Scope: apenas `build_rbs` e um novo método de ~2 linhas
- Sem risk de regressão — é adição pura
- Melhora a experiência em views/forms Rails

## Parte 2: Enumerize em módulos e concerns

### Problema

A gem enumerize permite definir atributos em módulos compartilhados:

```ruby
# app/models/concerns/publishable.rb
module Publishable
  extend ActiveSupport::Concern
  extend Enumerize

  enumerize :publish_status, in: [:draft, :published, :archived], predicates: true
end

# app/models/article.rb
class Article < ApplicationRecord
  include Publishable
end

# app/models/page.rb
class Page < ApplicationRecord
  include Publishable
end
```

O gerador atual tem duas limitações:

1. **`generate_all` só escaneia `app/models/**/*.rb`** — concerns em `app/models/concerns/` são escaneados, mas `extract_class_info` só procura `ClassNode`, ignorando `ModuleNode`.
2. **As classes que incluem o módulo não recebem os métodos** — o RBS gerado para `Article` e `Page` não vai incluir `publish_status`, `publish_status=`, etc.

### Estratégia em duas fases

#### Fase A: Gerar RBS para o módulo (escopo desta task)

Quando o gerador encontra um `ModuleNode` com chamadas `enumerize`, gerar um arquivo RBS para o módulo com os métodos de instância, predicados, etc.

**RBS esperado para o módulo**:

```rbs
# sig/rbs_enumerize/app/models/concerns/publishable.rbs
# Generated by rbs_infer (enumerize)

class ::Publishable::EnumerizePublish_statusValue < Enumerize::Value
  def draft?: () -> bool
  def published?: () -> bool
  def archived?: () -> bool
end

module ::Publishable
  def publish_status: () -> Publishable::EnumerizePublish_statusValue?
  def publish_status=: (String | Symbol | nil) -> (String | Symbol | nil)
  def publish_status_text: () -> String?
  def publish_status_value: () -> String?

  def draft?: () -> bool
  def published?: () -> bool
  def archived?: () -> bool
end
```

Quando `Article` e `Page` fazem `include Publishable`, o type checker (Steep) propaga naturalmente os métodos de instância do módulo para as classes.

**Nota sobre scopes**: Scopes (`scope: :shallow`, `scope: true`) são class methods — eles não propagam via `include` de módulo no RBS. Para módulos, as assinaturas de scope **não devem ser geradas** ou devem ser geradas num bloco `ClassMethods` separado se for um `ActiveSupport::Concern`. Isso é um detalhe de implementação a decidir.

#### Fase B: Propagação explícita para classes (futuro, fora desta task)

Para cenários onde Steep não resolve corretamente a inclusão do módulo, ou para scopes, pode ser necessário gerar RBS diretamente nas classes consumidoras. Isso requer um source index que rastreie quais classes incluem quais módulos. **Adiado para iteração futura.**

### Implementação — Fase A

#### 1. Modificar `extract_class_info` para aceitar `ModuleNode`

```ruby
def extract_class_info(node)
  case node
  when Prism::ProgramNode
    extract_class_info(node.statements)
  when Prism::StatementsNode
    node.body.each do |child|
      info = extract_class_info(child)
      return info if info
    end
    nil
  when Prism::ClassNode
    { name: constant_path_to_string(node.constant_path), type: :class }
  when Prism::ModuleNode                                          # ← novo
    { name: constant_path_to_string(node.constant_path), type: :module }
  else
    nil
  end
end
```

#### 2. Modificar `build_rbs` para distinguir `class` vs `module`

```ruby
def build_rbs(class_info, enumerize_calls)
  class_name = class_info[:name]
  is_module = class_info[:type] == :module

  # ... existente (value classes, etc.) ...

  if is_module
    lines << "module ::#{class_name}"
  else
    lines << "class ::#{class_name}"
    lines << "  extend Enumerize::ActiveRecordSupport"
  end

  enumerize_calls.each do |call|
    lines.concat(build_instance_methods(call))
    lines.concat(build_predicate_methods(call))
    # Scopes são class methods — não gerar para módulos simples
    lines.concat(build_scope_methods(call)) unless is_module
    lines.concat(build_class_attribute_accessor(call)) unless is_module
  end

  lines << "end"
  # ...
end
```

#### 3. Propagar `class_info` em `generate_for_model`

Atualizar para passar o `class_info` completo (com `:type`) para `build_rbs` em vez de apenas o nome:

```ruby
def generate_for_model(model_path)
  source = File.read(model_path)
  result = Prism.parse(source)

  class_info = extract_class_info(result.value)
  return unless class_info

  enumerize_calls = extract_enumerize_calls(result.value)
  return if enumerize_calls.empty?

  rbs = build_rbs(class_info, enumerize_calls)  # ← passa class_info inteiro
  # ... resto igual ...
end
```

#### 4. Ajustar `generate_all` para escanear concerns

O glob `app/models/**/*.rb` já cobre `app/models/concerns/**/*.rb` por padrão. Se concerns estiverem em outro local (ex: `app/concerns/`), expandir o glob:

```ruby
def generate_all
  patterns = [
    File.join(app_dir, "app/models/**/*.rb"),
    File.join(app_dir, "app/concerns/**/*.rb"),  # ← caso comum em alguns projetos
  ]

  patterns.flat_map { |p| Dir.glob(p) }.uniq.sort.each do |model_path|
    generate_for_model(model_path)
  end
end
```

### Testes necessários

#### Fixtures a criar

Criar um concern de teste no dummy app:

```ruby
# spec/dummy/app/models/concerns/categorizable.rb
module Categorizable
  extend ActiveSupport::Concern
  extend Enumerize

  enumerize :item_type, in: [:article, :video, :podcast], predicates: { prefix: true }
end
```

E uma model que o inclui:

```ruby
# spec/dummy/app/models/media_item.rb
class MediaItem < ApplicationRecord
  include Categorizable
end
```

#### Expectativas

1. Gerar `sig/rbs_enumerize/app/models/concerns/categorizable.rbs`:
   - **Module** `::Categorizable` (não class)
   - Value class `::Categorizable::EnumerizeItem_typeValue`
   - Instance methods (`item_type`, `item_type=`, `item_type_text`, `item_type_value`)
   - Predicados com prefixo (`item_type_article?`, `item_type_video?`, `item_type_podcast?`)
   - **Sem scopes** (módulo, não classe AR)

2. Steep aceita `media_item.item_type.article?` sem erros

### Edge cases a considerar

| Caso | Decisão |
|------|---------|
| Módulo com `scope:` option | Ignorar scopes para módulos (são class methods de AR) |
| Módulo sem `extend Enumerize` (herda via include) | Fora de escopo — exigiria análise de inclusão transitiva |
| Módulo com `extend ActiveSupport::Concern` + `included` block com enumerize | O enumerize dentro do `included` block está no `class_eval` da classe, não do módulo. O gerador estático não diferencia isso. Documentar como limitação. |
| Nesting: `module Post::Publishable` | Funciona — `constant_path_to_string` já resolve `ConstantPathNode` |
