# ERB View Type Inference

## The problem

ERB templates are parsed as Ruby by Steep, but their self type is unknown — `self` inside
a view could be anything. Without a hint, Steep cannot resolve instance variables like
`@post` or helper method calls:

```erb
<h1><%= @post.title %></h1>  <!-- ← Steep doesn't know what @post is -->
<%= truncate(@post.body) %>   <!-- ← or that truncate comes from a helper -->
```

The solution is a `# @type self:` annotation pointing to a generated class that declares
all instance variables, locals, and included helpers for that specific view.

---

## The solution: two cooperating layers

The ERB convention is split between the **Steep fork** (which handles type checking at
`steep check` time) and **rbs_infer** (which generates the RBS definitions that Steep
reads):

```
┌──────────────────────────────────────────────────────────────────┐
│  rbs_infer                                                       │
│                                                                  │
│  ErbConventionGenerator                                          │
│  ├── ControllerAnalyzer   → extracts @ivar types from actions    │
│  ├── PartialLocalsCollector → infers locals from render calls    │
│  └── writes sig/rbs_infer_erb/app/views/**/*.rbs                 │
│      e.g. class ERBPostsShow                                     │
│             @post: Post                                          │
│             @comments: Comment::ActiveRecord_Relation            │
│             include PostsHelper                                  │
│             include ActionViewContext                             │
│           end                                                    │
│                                                                  │
│  ErbCallerResolver                                               │
│  └── scans ERB files as call sites when inferring helper types   │
└───────────────────────────────┬──────────────────────────────────┘
                                │ RBS consumed by
                                ▼
┌──────────────────────────────────────────────────────────────────┐
│  Steep fork                                                      │
│                                                                  │
│  Source.parse(source_code, path:, factory:)           [source.rb]│
│  └── if path ends with .erb:                                     │
│        source_code = extract_ruby_from_erb(source_code)          │
│        if STEEP_ERB_CONVENTION && class_name = resolve(path):    │
│          source_code += "\n# @type self: #{class_name}"          │
│                                                                  │
│  ErbSelfTypeResolver.resolve(path)     [erb_self_type_resolver.rb]│
│  └── maps file path → ERB class name                             │
│      app/views/posts/show.html.erb    → ERBPostsShow             │
│      app/views/posts/_form.html.erb   → ERBPartialPostsForm      │
└──────────────────────────────────────────────────────────────────┘
```

At `steep check` time:
1. Steep parses the `.erb` file, extracts Ruby from it with `Herb.extract_ruby`
2. Appends `# @type self: ERBPostsShow` (derived from the path)
3. The parsed Ruby now has a known self type
4. Steep resolves `@post` against the `ERBPostsShow` RBS definition generated by `rbs_infer`

---

## Enabling the feature

```ruby
# Steepfile
ENV["STEEP_ERB_CONVENTION"] = "1"

target :app do
  check "app/views/**/*.erb"
  signature "sig"           # where rbs_infer writes the generated RBS
end
```

Generate the RBS files before running Steep:

```sh
bundle exec rake rbs_infer:erb:all
```

## Class name convention

`ErbSelfTypeResolver` (Steep fork) and `ErbConventionGenerator` (rbs_infer) apply the
same naming rule so both sides always agree:

| Path | Class name |
|---|---|
| `app/views/posts/show.html.erb` | `ERBPostsShow` |
| `app/views/posts/index.html.erb` | `ERBPostsIndex` |
| `app/views/posts/_form.html.erb` | `ERBPartialPostsForm` |
| `app/views/admin/posts/show.html.erb` | `ERBAdminPostsShow` |
| `app/views/layouts/application.html.erb` | `ERBLayoutsApplication` |
| `app/views/user_mailer/welcome.html.erb` | `ERBUserMailerWelcome` |
| `app/views/posts/create.turbo_stream.erb` | `ERBPostsCreate` |

Algorithm:
1. Strip `app/views/` prefix
2. Strip `.html.erb` / `.turbo_stream.erb` suffix
3. If the filename starts with `_`, it's a partial — strip the `_` and use `ERBPartial` prefix
4. Camelize each path segment and join: `posts/show` → `ERBPostsShow`

## What rbs_infer generates

### Regular view

For `app/views/posts/show.html.erb`, `rbs_infer` finds the matching action
`PostsController#show`, extracts all instance variables it assigns, and writes:

```ruby
# sig/rbs_infer_erb/app/views/posts/show.rbs
class ERBPostsShow
  @post: Post
  @comments: Comment::ActiveRecord_Relation
  def params: () -> ActionController::Parameters
  include PostsHelper
  include ActionViewContext
end
```

### Partial

For `app/views/posts/_form.html.erb`, there is no controller action. Instead, `rbs_infer`
scans every call to `render "posts/form"` or `render partial: "posts/form", locals: {…}`
across all controllers and ERB files, collects the locals passed, and writes:

```ruby
# sig/rbs_infer_erb/app/views/posts/_form.rbs
class ERBPartialPostsForm
  attr_reader post: Post
  def params: () -> ActionController::Parameters
  include ActionViewContext
end
```

### Layout

Layouts have no controller action and no render-site locals, so they only get `params`
and the `ActionViewContext` include:

```ruby
# sig/rbs_infer_erb/app/views/layouts/application.rbs
class ERBLayoutsApplication
  def params: () -> ActionController::Parameters
  include ActionViewContext
end
```

## How instance variable types are resolved

`ControllerAnalyzer` runs `rbs_infer`'s own `Analyzer` against the controller file, then
filters the inferred instance variables down to those written by the target action and its
`before_action` callbacks:

```
PostsController#show
  before_action :set_post  → assigns @post: Post
  show action              → assigns @comments: Comment::ActiveRecord_Relation
```

Only variables reachable from that specific action are included in the view's RBS, not
every ivar in the controller.

---

## How partial local types are resolved

`PartialLocalsCollector` does a two-phase scan before generating any file:

1. **Scan phase** — walks every `.rb` and `.erb` file looking for `render` calls:
   ```ruby
   render partial: "posts/form", locals: { post: @post }
   render "posts/summary", post: @post   # shorthand
   ```
2. **Type resolution** — for each local, looks up `@post`'s type from the controller's
   inferred RBS (using `ErbCallerResolver` internally)

The result is a map `"posts/form" → { post: "Post" }` that `ErbConventionGenerator` uses
when generating the partial's RBS.

---

## Helper method type inference (ErbCallerResolver)

When `rbs_infer` infers types for a helper module (e.g. `PostsHelper`), it needs to know
what arguments the views actually pass to each helper method. `ErbCallerResolver` bridges
this gap:

```ruby
# rails_dummy_spec.rb
erb_resolver = RbsInfer::Extensions::Rails::ErbCallerResolver.new(
  app_dir: Dir.pwd,
  source_files: source_files
)
assert_snapshot("helpers/application_helper",
  target_class: "ApplicationHelper",
  target_file: "app/helpers/application_helper.rb",
  extra_caller_sources: erb_resolver
)
```

`ErbCallerResolver` implements the `extra_caller_sources` interface: the `Analyzer` calls
it alongside the regular Ruby source files, and it provides the ERB views as additional
call sites. This lets `rbs_infer` infer parameter types for helper methods from how the
views use them, without any Rails-specific logic inside the core `Analyzer`.

## Comparison with the module convention

Both conventions follow the same core pattern — derive a name from the file path, inject
a `@type self:` annotation in memory, consume a generated RBS definition — but differ
in scope and direction:

| | ERB convention | Module convention |
|---|---|---|
| Env var | `STEEP_ERB_CONVENTION` | `STEEP_MODULE_CONVENTION` |
| Annotation injected | `# @type self: ERBPostsShow` | `# @type self:` + `# @type instance:` |
| RBS generated by | `ErbConventionGenerator` (writes files) | Not generated — reads existing model RBS |
| Name derivation | `app/views/posts/show.html.erb` → `ERBPostsShow` | `app/models/post/notifiable.rb` → `Post::Notifiable` |
| Including class | N/A (self type is the ERB class itself) | Derived from module namespace (`Post`) |
| Extra rbs_infer step | `Analyzer` does not need to re-apply | `Analyzer` re-applies resolver before `Prism.parse` |
